# Phase 44 Context: Validator-backed Trust Surfaces

**Milestone:** v1.5 Validation and Trust Surfaces
**Objective:** Strengthen the evidence and support surface around produced PDFs without pretending to offer universal compliance.

## Discussion State

We have course-corrected the milestone away from out-of-scope product concerns (auth, feature flags) back to the library's pure core DNA. The goal is to prove that the PDFs generated by Rendro are structurally sound and meet expected standards, and to provide a machine-readable support matrix for these claims.

Through the `grill-me` sequential interview, we established the following alignment for Phase 44:

### 1. Validation Approach
**Decision:** External Binary Adapter.
**Rationale:** Building a full PDF compliance parser in pure Elixir is a massive undertaking that distracts from the core rendering engine. Shelling out to an established tool provides immediate trust and utility.

### 2. Target Validator
**Decision:** `pdfinfo` (from Poppler).
**Rationale:** It proves basic structural integrity (that our Elixir output isn't corrupt) and is easy to run in CI, before tackling heavier, stricter tools like VeraPDF.

### 3. Integration Shape
**Decision:** `Rendro.Adapters.Poppler` module.
**Rationale:** A dedicated adapter exposing `validate/1` allows users to cleanly assert valid PDFs in their own ExUnit test suites, empowering them to verify their generated documents locally.

### 4. Missing Dependency Handling
**Decision:** Runtime Error Tuple.
**Rationale:** Returning `{:error, {:missing_executable, "pdfinfo"}}` allows users to gracefully skip validation in environments without Poppler (like certain CI runners) rather than crashing their test suites abruptly.
