  <behavior>
    - Test 1: `measure_block/3` for `%Rendro.Table{}` normalizes lightweight lists-of-lists and explicit `%Rendro.Row{}`/`%Rendro.Cell{}` structs into a 2D Grid matrix.
    - Test 2: Spanning cells (colspan/rowspan) generate continuation grid slots with an `is_continuation: true` flag.
    - Test 3: Tables that result in a matrix exceeding the maximum cell limit return `{:error, :grid_too_large}`.
  </behavior>
  <action>
    Refactor `measure_table_rows` in `lib/rendro/pipeline/measure.ex`. Instead of keeping tables as nested trees of `Row -> Cell`, project the layout into a 2D list-of-lists grid structure.
    First, normalize the table's `:rows` field. The input might be the lightweight DSL (a list of lists of strings/blocks) or explicit `%Rendro.Row{}` and `%Rendro.Cell{}` structs. Use pattern matching to normalize raw list elements into these structs (e.g., mapping `[ [cell1, cell2] ]` into `[%Rendro.Row{cells: [%Rendro.Cell{content: cell1}, %Rendro.Cell{content: cell2}]}]`).
    Then, project this normalized list of `Row` structs into a 2D grid matrix. If a cell spans multiple rows/cols, it populates multiple indices in the grid. The primary slot holds the content, and spanned slots hold a continuation marker (e.g., `%{is_continuation: true, ref: primary_cell_id}`).
    Enforce a hard limit on matrix size during projection to prevent OOM DoS (Threat T-37-01). If the calculated grid dimensions (rows * cols) exceed a reasonable maximum (e.g., 100,000 cells), halt projection and return `{:error, :grid_too_large}`.
    Store this projected grid onto a new private field on the returned `Table` struct (e.g. `_grid_layout`) so it can be consumed by the Paginate phase.
  </action>

    <automated>mix test test/rendro/pipeline/measure_test.exs</automated>
