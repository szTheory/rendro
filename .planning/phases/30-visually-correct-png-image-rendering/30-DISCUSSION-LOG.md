### PNG Decoding and Encoding Strategy
- **Assumption:** The PDF writer should leverage PDF's native FlateDecode with `/DecodeParms` for RGB/Gray PNGs, and must manually split the alpha channel into a separate `/SMask` stream for RGBA PNGs, replacing any raw-file-byte pass-through.
  - **Why this way:** `lib/rendro/pdf/writer.ex` demonstrates this approach by utilizing `PNG.decode_pixels/1` and `PNG.split_alpha/2` (from `lib/rendro/pdf/png.ex`) for color types 4 and 6 to create two separate zlib-compressed streams (Color and `/SMask`). For color types 0, 2, and 3, it passes the raw `png.idat` chunks but correctly provides a `DecodeParms` dictionary with predictor and color metadata.
  - **If wrong:** If raw PNG container bytes are written or `DecodeParms` are omitted for filtered scanlines, PDF readers will attempt to inflate the data as standard zlib, fail, and silently skip drawing the image, leaving blank spaces exactly as caught in earlier UATs.
  - **Confidence:** Confident

### Visual Rasterization Regression Test Implementation
- **Assumption:** The new regression test class will rely on shelling out to `pdftoppm` to generate a rasterized PNG of the PDF and perform targeted pixel color assertions to prove the image paints.
  - **Why this way:** The phase context dictates adding a "rasterize-and-decode regression class." Furthermore, `lib/mix/tasks/rendro.visual_uat.ex` already successfully shells out to `System.cmd("pdftoppm", ["-png", "-r", "200", ...])` to verify output for the manual UAT task. Existing tests in `test/rendro/recipes/branded_invoice_test.exs` and `test/rendro/pdf/writer_test.exs` only perform structural byte-level assertions (e.g., `assert pdf =~ "/IM_COMPANY_LOGO"`), making them the ideal targets to pair with a visual assertion.
  - **If wrong:** Relying solely on byte-substring tests or manual Claude Vision checks creates automated blind spots, allowing broken image encodings to pass the CI suite as long as the PDF syntax is valid.
  - **Confidence:** Confident

### CI Toolchain Updates for Poppler
- **Assumption:** The GitHub Actions workflow must be explicitly updated to install the `poppler-utils` package before running the test suite.
  - **Why this way:** `.github/workflows/ci.yml` currently runs on `ubuntu-latest` and only installs standard Elixir/Beam dependencies. However, `lib/mix/tasks/rendro.visual_uat.ex` explicitly notes that `apt install poppler-utils` is required to provide the `pdftoppm` executable on Debian-based systems.
  - **If wrong:** The new visual regression tests will pass locally for developers with poppler installed via Homebrew, but will consistently fail the CI pipeline when `System.cmd("pdftoppm", ...)` encounters an `enoent` (executable not found) error.
  - **Confidence:** Confident
