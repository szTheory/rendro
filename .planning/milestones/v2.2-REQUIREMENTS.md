# Requirements: Rendro v2.2 Long-Lived Signatures & Compliance Evidence

**Defined:** 2026-05-07
**Status:** Planned
**Core Value:** Phoenix teams can generate reliable, auditable, deterministic PDFs from Elixir data/components, with clear pagination behavior and production-grade observability.

## Milestone Goal

Turn the proof-backed signing seam into a truthful long-lived-signature story by adding artifact-first timestamp and revocation evidence support, validator-backed posture classification, and exact compliance-language boundaries.

## In-Scope Requirements

### Long-Lived Artifact Semantics

- [x] **SIGN-07**: Engineers can extend a supported signed `%Rendro.Artifact{}` through one explicit artifact-stage API that adds timestamp and revocation evidence without changing `Rendro.render/2` or bare `Rendro.Sign.sign/2` semantics.
- [x] **SIGN-08**: Long-lived augmentation rejects unsigned artifacts, unsupported signature state, malformed adapter configuration, and secret-bearing runtime failures with typed, redacted errors.
- [x] **SIGN-09**: Long-lived artifacts expose explicit metadata describing timestamp, revocation, and compliance-evidence posture without persisting keys, passphrases, or raw tool output.

### Adapter and Validator Boundaries

- [x] **ADAPT-07**: Rendro ships a first-party optional adapter path that can add document timestamp and revocation evidence over the supported signed-artifact seam.
- [x] **ADAPT-08**: Rendro ships a validator-backed inspection path that reports cryptographic integrity, timestamp presence, revocation evidence presence, and narrow compliance posture as distinct signals.
- [x] **ADAPT-09**: Rendro includes a live proof lane that produces a representative long-lived signed artifact and verifies the resulting posture through the supported validator path.

### Truthful Compliance Boundaries

- [x] **TRUST-07**: `priv/support_matrix.json` publishes long-lived-signature and narrow compliance support separately from bare signing, signer identity trust, viewer behavior, and broad compliance narratives.
- [x] **TRUST-08**: Public docs explain timestamp/revocation evidence, certificate trust, viewer behavior, and narrow compliance claims as distinct boundaries and keep unsupported rows explicit.
- [x] **TRUST-09**: Verification artifacts prove the exact supported long-lived/compliance path without implying blanket PDF/A, viewer support, multi-signature workflows, or generic regulatory approval. Closed by `.planning/phases/67-verification-and-milestone-closure/67-VERIFICATION.md`; operational branch-protection confirmation remains tracked separately under `ADAPT-09`.

## Future Requirements

- Proof-backed viewer promotion for forms, protection, signing preparation, and signed artifacts across Adobe Acrobat Reader, Chrome PDFium, PDF.js, and Apple Preview
- Additional signing and long-lived-signature adapters beyond the first proof-backed path
- Multi-signature workflows, counter-signatures, and signer-identity orchestration
- Broader compliance profiles, trust-store management, HSM integration, and regulatory packaging
- Batteries-included onboarding workflows, richer canonical recipes, and stronger out-of-the-box adoption guides
- Global text shaping, RTL support, and broader script coverage with explicit proof surfaces

## Out of Scope

- In-core key custody, certificate-store management, or HSM orchestration
- Blanket compliance branding or vague “enterprise signing” narratives without artifact-level proof
- Viewer promotion without exact recorded per-viewer evidence
- Multi-signature workflows, signer routing/orchestration, or hosted approval flows
- HTML/CSS/browser-renderer parity
