# Rendro v1.5 Architectural Research & Recommendations

## 1. Widow/Orphan Layout Controls
**Goal:** Multi-line text blocks breaking cleanly across page boundaries without leaving solitary lines.

### Approach A: Greedy Layout with Backtracking (Post-Measurement)
*   **Pros:** Conceptually simple.
*   **Cons:** State management is tricky. Backtracking in a pure functional pipeline means unwinding an accumulator, which can cascade.

### Approach B: Predictive Fragmentation (`split/2` Protocol) (RECOMMENDED)
*   **Pros:** One-pass pagination. The pagination engine asks the block "what fits in H?", and the block handles its own internal constraints.
*   **Cons:** Slightly complex constraint solving inside the `Text` module.

### Ecosystem Lessons & Elixir Idioms
*   **Lessons:** `pdfmake` uses a top-down greedy approach with keep-together flags, which is pragmatic compared to complex algorithms like TeX's.
*   **Idiomatic Elixir:** Leverage a `Rendro.Fragmentable` protocol. `def fragment(node, available_height)` returns `{:ok, fit_node, overflow_node}`.

## 2. Advanced Table Continuation & Explicit Fragmentation
**Goal:** Multi-page tables strictly following explicit cell fragmentation rules.

### Approach A: Atomic Row-Based Fragmentation
*   **Pros:** Easy to calculate bounding boxes and draw borders.
*   **Cons:** Creates massive whitespace gaps if a single cell contains a huge text block.

### Approach B: Cell-Level (Sub-Row) Fragmentation (RECOMMENDED)
*   **Pros:** Eliminates wasted whitespace. Handles arbitrarily large data payloads seamlessly.
*   **Cons:** Drawing borders across page breaks is complex.

### Ecosystem Lessons & Elixir Idioms
*   **Lessons:** iText (Java) is the gold standard here. ReportLab struggles with deep nested tables breaking gracefully.
*   **Idiomatic Elixir:** Treat Tables as a matrix reduction. We map over cells and invoke `Fragmentable.fragment/2`. A `TableContinuity` struct can hold the "split state" to ensure borders are drawn correctly on subsequent pages.

## 3. Nested Layout Structures
**Goal:** Tables nested inside other blocks or tables aggregating heights correctly and fragmenting cleanly.

### Approach A: Bottom-Up Absolute Flattening
*   **Pros:** Rendering phase is incredibly fast and dumb.
*   **Cons:** Flattening destroys the logical grouping required to figure out where a page break can safely occur.

### Approach B: Recursive AST Fragmentation (Virtual DOM Approach) (RECOMMENDED)
*   **Pros:** Structurally sound. Preserves semantics. Allows for complex keep-together behaviors.
*   **Cons:** Tree traversal can be CPU intensive.

### Ecosystem Lessons & Elixir Idioms
*   **Lessons:** `pdfmake` excels here by maintaining a declarative layout tree that resolves bottom-up (measurement) and paginates top-down.
*   **Idiomatic Elixir:** A tree traversal using recursion and pattern matching on structs is extremely performant on the BEAM.

---

## 4. Structural PDF Validation & Stress Test
**Goal:** Native structural validation passes to prove layout soundness without relying on heavy external C-binaries.

### Approach A: Post-Serialization Parser (Native PDF Reader)
*   **Pros:** Validates the actual byte output. Extremely high confidence.
*   **Cons:** Writing a robust PDF parser is a massive undertaking.

### Approach B: Pre-Serialization AST Invariant Checker (Advisory Auditing) (RECOMMENDED)
*   **Pros:** Pure Elixir, fast, and deterministic. Catches 99% of structural bugs before they become corrupted PDFs.
*   **Cons:** Doesn't test the actual binary serialization phase.

### Ecosystem Lessons & Elixir Idioms
*   **Lessons:** HexaPDF implements both reading and writing, meaning its test suite parses its own output. For Rendro, structural invariant checking is the next best thing without building a full reader.
*   **Idiomatic Elixir:** Implement as an optional stage `Rendro.Pipeline.Validate` that takes the final `Document` state and runs invariant checks, returning `{:ok, doc}` or `{:error, diagnostics}`.

---

## Cohesive Synthesis & Final Recommendations
To build a bulletproof, pure-Elixir deterministic engine with a fantastic DX, Rendro should adopt the **Recursive Virtual DOM** architecture with a universal **`Fragmentable` Protocol**.

### The Unified Architecture: "Measure Bottom-Up, Fragment Top-Down"
1.  **Measurement Phase:** Traverse the nested AST bottom-up. Every node calculates intrinsic width/height.
2.  **Pagination Phase:** Traverse the `MeasuredAST` top-down. Provide each node with `available_capacity`. Implement `Rendro.Fragmentable.fragment(node, capacity) :: {:ok, fit_node, overflow_node}`.
    *   **Text (Slice 1):** Shifts lines to `overflow_node` if splitting violates widow/orphan rules.
    *   **Tables (Slice 2 & 3):** Asks Rows to fragment, which asks Cells. Deeply nested tables naturally fragment via recursive protocol calls.
3.  **Rendering Phase:** Translates paginated AST into PDF objects.
4.  **Validation Phase (Slice 4):** Invariant checker runs over the PDF object tree right before binary serialization.

**Why this is perfect:**
*   **Determinism:** Layout and fragmentation are pure functional transformations of a data structure.
*   **DX:** The developer never thinks about coordinates. They declare a nested tree and define constraints (`widows: 2`). The protocol handles the rest.
*   **Idiomatic:** Relies entirely on structs, protocols, and functional reduction.
