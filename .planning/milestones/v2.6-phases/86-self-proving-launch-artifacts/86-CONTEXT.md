# Phase 86: Self-Proving Launch Artifacts - Context

**Gathered:** 2026-06-11
**Status:** Ready for planning

<domain>
## Phase Boundary

Deliver the Phase 86 `GAL-01..03` launch-proof artifacts:

1. A rendered visual recipe gallery for all five canonical recipes (Invoice, BrandedInvoice, Statement, Receipt/Report, Certificate) in README and HexDocs.
2. A Rendro-generated `manual.pdf` with machine-published SHA-256 and CI verification.
3. Presentation that follows the Rendro brand book: calm, document-native, accessible, honest, and proof-backed.

This phase does not add new user-facing document capabilities beyond what Phases 84 and 85 already shipped. It packages those capabilities into launch artifacts that make the existing engine visible and auditable.

</domain>

<decisions>
## Implementation Decisions

### Showcase Fixture Posture
- **D-01:** Use a **hybrid fixture posture**. README/guide code examples stay canonical zero-to-one defaults; gallery/manual source PDFs are deterministic launch fixtures under `Rendro.LaunchArtifacts` that may opt into subtle Phase-84 polish. This preserves least surprise while making the launch gallery visually credible.
- **D-02:** Do **not** change global table defaults or canonical recipe defaults for screenshots. Phase 84 locked table borders/rules/header bands as opt-in; Phase 86 must not silently turn optional polish into default recipe behavior.
- **D-03:** Gallery/manual fixture styling should be quiet and brand-aligned: table `borders: [:outer, :rows]`, warm header band (`{247, 243, 234}`), line color near brand `line-300` (`{216, 210, 195}`), and modest stroke width (about `0.6`). Avoid heavy `borders: :all` on long Statement/Receipt tables; use dense grid styling only in a small manual proof if needed.
- **D-04:** Certificate should keep using explicit `border: true` in launch fixtures. It is a public recipe option, exercises the Phase-84 Path-backed frame, and matches the gallery caption.
- **D-05:** Public copy must call the gallery output **curated deterministic recipe fixtures** or equivalent wording. Do not imply tuned launch fixtures are the default recipe output.

### Self-Rendered Manual Depth
- **D-06:** Ship a compact self-rendered `manual.pdf` as a proof artifact, not a full Prawn-style reference manual. Target roughly 8-12 pages: enough to prove the pipeline and show the recipes, not enough to duplicate HexDocs.
- **D-07:** Manual structure should be layered:
  - positioning/proof cover ("Native PDF layout for Elixir", generated by Rendro itself, SHA published externally);
  - fit check and boundaries (good fits, when HTML-to-PDF is better, complex-script boundary, no PDF/A/PDF/UA claim, pdfium-render is not GUI-viewer proof);
  - one page per recipe with rendered image, JTBD caption, module name, and manifest/hash pointer;
  - determinism proof (manifest, source PDF hashes, PNG hashes, renderer metadata, `mix rendro.launch_artifacts.check`);
  - Path/page-number proof (certificate frame or diagram plus running footer).
- **D-08:** HexDocs, `guides/recipes.md`, and Phase 87 Livebook carry teaching depth. `manual.pdf` points readers there; it does not become a second API reference.

### Verification Split
- **D-09:** Required docs-contract checks cover deterministic/static proof only: manifest shape, exact five gallery IDs, committed asset hashes versus manifest, regenerated source-PDF hashes, regenerated `manual.pdf` hash, generated README/guide blocks, package inclusion, alt text, and overclaim guards.
- **D-10:** PNG regeneration stays in the advisory `raster-advisory` lane using pinned pdfium. The advisory lane should run the launch artifact check that regenerates PNGs, validates dimensions, and verifies renderer-version metadata.
- **D-11:** Required `mix ci` must not require `pdfium-cli`, download external raster tooling, or call a raster regeneration path. That would violate the Phase-85 advisory-lane boundary and make optional external tooling block core engine merges.
- **D-12:** Public wording must be precise: source PDFs/manual/manifest are required docs-contract checked; PNG rasters are regenerated and hash-checked in the advisory pdfium lane. Never upgrade `pdfium-render` evidence into GUI-viewer proof.

### README / HexDocs Presentation
- **D-13:** Keep the README first screen concise and ecosystem-idiomatic: badges, tagline, one positioning sentence, feature bullets, then rendered gallery/manual hash block. Do not put a full hero-style composite before the reader knows what Rendro is.
- **D-14:** Tighten the opening copy to the brand-book direction: "Native PDF layout for Elixir" and "open-source, Elixir-native PDF layout library for Phoenix teams without Chrome" language.
- **D-15:** Use the README gallery as a quick five-thumbnail proof. Use `guides/recipes.md` as the richer presentation with larger images, captions, source PDF hashes, PNG hashes, and manual SHA.
- **D-16:** Presentation details: white document sheets, subtle ExDoc border/shadow styling, linked full-size PNGs, descriptive alt text, no browser/printer/prepress cues, no composite marketing hero in README, and no dark-mode duplicate assets unless a future single hero image needs them.

### the agent's Discretion
- Exact implementation shape for applying table polish to launch fixtures is left to planning/execution. Acceptable paths include using the three-rung recipe escape hatch to construct launch-specific sections, or adding a narrow inert-default recipe option if that is clearly lower surprise. Do not add a broad public styling API only for screenshots.
- Exact fixture data values, image dimensions, and manual page count are flexible as long as all five recipes are shown, the artifacts remain deterministic, and the results follow the brand book.
- Exact module/task names may follow existing scaffolding if retained; preserve the core decision that generation and checking are one explicit launch-artifact workflow.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `Rendro.Adapters.Pdfium.render/2` renders PDFs to PNGs with `dpi:` and `pages:` options. Use it only in advisory raster checks or local generation, not required `mix ci`.
- `Rendro.LaunchArtifacts` scaffolding already renders five gallery entries, writes `assets/rendro/artifacts.json`, writes `manual.pdf`, and updates README/recipes-guide generated blocks. Treat this as existing user/agent work and modify carefully.
- `Rendro.Recipes.*` modules expose `document/2`, `page_template/1`, and `sections/2`; this escape hatch can support launch-specific fixture assembly without changing default recipe behavior.
- `Rendro.Recipes.Certificate.document(data, border: true)` already exercises the Phase-84 Path-backed frame in a user-visible way.
- `scripts/verify_docs.exs` and docs-contract tests are the established mechanism for making documentation claims executable.

### Established Patterns
- Deterministic proof belongs in required docs-contract lanes; external-tool/raster proof stays advisory unless already part of the four required engine lanes.
- Package contents are explicit in `mix.exs`; public assets must be listed intentionally.
- README/HexDocs copy must be bounded by `priv/support_matrix.json` and current proof artifacts. Avoid blanket viewer, compliance, HTML/CSS, and complex-script claims.
- ExDoc assets should use simple relative paths under `assets/`, with styling centralized in `mix.exs` rather than bespoke Markdown hero HTML.
- Errors and drift messages should be errors-as-product: tell the maintainer what drifted and which generation/check command to run.

### Integration Points
- `Rendro.LaunchArtifacts.generate/1` should own writing PNGs, manual PDF, manifest, and generated docs blocks.
- `Rendro.LaunchArtifacts.static_contract_errors/0` should remain safe for required docs-contract tests with no pdfium dependency.
- `mix rendro.launch_artifacts.check` can require pdfium when run in `raster-advisory` and should report PNG drift, dimension drift, and renderer-version drift.
- README and `guides/recipes.md` generated blocks should be source-of-truth-rendered from the manifest so hashes and copy cannot drift independently.

</code_context>

<specifics>
## Specific Ideas

- The launch artifact set should borrow Prawn's self-demonstrating manual idea without copying its full manual breadth. Prawn proves the power of self-generated docs; Rendro's better Phase-86 fit is a compact proof PDF plus searchable, docs-contract-checked HexDocs.
- Typst Universe validates that visual galleries drive adoption, but Rendro should keep the gallery tied to real recipe fixtures and artifact hashes rather than a free-form template marketplace.
- Phoenix/Ecto/Oban/Req ecosystem docs set the README expectation: explain the product, show concise examples/proof, and push deeper learning into guides/module docs. Avoid overbuilt README marketing layout.
- Current generated invoice/statement/receipt preview images are truthful but visually sparse. The planned launch fixtures should visibly exercise Phase-84 polish while making the optional nature explicit.
- Good microcopy phrases:
  - "Rendered by Rendro from deterministic recipe fixtures."
  - "Source PDFs and the manual are checked by the required docs contract; PNG rasters are regenerated in the advisory pdfium lane."
  - "Pdfium rasters are render proof, not GUI-viewer proof."

</specifics>
