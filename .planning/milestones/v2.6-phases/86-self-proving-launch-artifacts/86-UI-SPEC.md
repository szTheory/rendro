# Phase 86 - UI Design Contract

> Visual and interaction contract for README, HexDocs recipe gallery, generated `manual.pdf`, rendered document previews, generated microcopy, alt text, and ExDoc light/dark image treatment. Generated by gsd-ui-researcher, pending gsd-ui-checker verification.

---

## Design System

This is not a web app UI phase. No shadcn, component registry, client-side framework, or interactive widget system is in scope.

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | none |
| Icon library | none |
| Primary rendering surfaces | GitHub README Markdown, HexDocs/ExDoc Markdown, generated `manual.pdf`, PNG document previews |
| Source of truth for generated blocks | `Rendro.LaunchArtifacts.readme_block/1` and `Rendro.LaunchArtifacts.recipes_block/1` |
| Source of truth for artifact metadata | `assets/rendro/artifacts.json` |

Contract:

- Generated README and recipes-guide blocks must be edited through `Rendro.LaunchArtifacts`; hand edits are drift and should fail the docs contract.
- README and HexDocs must use simple Markdown/HTML that works in GitHub and ExDoc without custom JavaScript.
- ExDoc styling may be centralized in `mix.exs`; do not add per-page bespoke HTML wrappers or layout scripts.

---

## README First Screen

The README first screen must stay ecosystem-idiomatic and concise. It is a proof surface, not a launch landing page.

Required order:

1. `# Rendro`
2. CI/Hex/HexDocs badges.
3. Tagline: `Native PDF layout for Elixir.`
4. First-mention sentence using this meaning: `Rendro is an open-source, Elixir-native PDF layout library for Phoenix teams that need reliable PDFs without Chrome.`
5. Short feature bullets focused on pure core, deterministic output, flow/pagination, tables, Phoenix fit, telemetry/diagnostics, and proof-backed boundaries.
6. Generated launch artifact block with five thumbnail previews and the `manual.pdf` SHA-256.
7. Deeper guides and API examples after the proof block.

Gallery placement:

- The gallery must appear before long API walkthroughs, after the positioning/features block.
- README gallery is a quick five-thumbnail proof, not the rich explanatory gallery.
- Thumbnail width: `150`.
- Thumbnail order: Invoice, Branded Invoice, Statement, Receipt / Report, Certificate.
- Every thumbnail must link to its full-size PNG with the same relative path used by the image `src`.

README anti-patterns:

- No hero composite before the reader knows what Rendro is.
- No browser-window, printer, Acrobat, prepress, viewer-SDK, stock-photo, or generic PDF-icon cues.
- No dark/blurred/atmospheric launch artwork.
- No claim that launch fixture styling is the canonical recipe default.

## HexDocs Recipes Gallery

`guides/recipes.md` is the richer presentation surface.

Required structure:

1. Opening copy must not say there are only four visual examples. Use wording such as `Rendro ships canonical recipes and five rendered gallery entries, including a branded invoice variant.`
2. Generated `## Rendered Gallery` block near the top of the guide.
3. One section per entry, in manifest order.
4. Linked image with width `320`.
5. Plain-language caption adjacent to each image.
6. Source PDF SHA-256 and PNG SHA-256 shown as selectable inline code.
7. `## Self-Rendered Manual` block after the gallery entries with a link to `assets/rendro/manual.pdf` and its SHA-256.

Gallery entry contract:

| Entry | Required visual proof |
|-------|-----------------------|
| Invoice | Header, date, table with launch-only row/outer rules, warm header band, footer note |
| Branded Invoice | Logo, embedded brand font, readable heading with no overlap or split letters, table polish |
| Statement | Multi-page statement proof via Page 1 footer, running balances, repeated/structured table header |
| Receipt / Report | Multi-page report proof via Page 1 footer, repeated table header, totals |
| Certificate | Landscape sheet, `border: true` Path-backed frame, recipient/body text, no hardcoded A4-only layout cue |

Visual acceptance:

- Current invoice-style sparseness is not launch-ready by itself; launch fixtures must visibly exercise Phase 84 table polish.
- Current branded-invoice heading overlap/split lettering is a GAL-03 blocker until regenerated and visually reviewed.
- Dense Statement and Receipt tables should use calm row/outer rules; avoid full cell grids on long tables.

## Manual PDF

The manual is a compact proof artifact, not a second API reference.

Required shape:

| Property | Contract |
|----------|----------|
| Page count | 8-12 pages |
| Page size | A4 portrait unless a deliberate change is recorded |
| Margins | 54pt outer page margin baseline |
| Footer | Running `Rendro manual - Page {{page_number}} of {{total_pages}}` footer on every page |
| Source | Generated by Rendro itself through `Rendro.LaunchArtifacts.render_manual_pdf/0` |
| Hash | SHA-256 published outside the PDF in README and recipes guide |

Required section hierarchy:

1. Cover: `Native PDF layout for Elixir`, generated-by-Rendro statement, external SHA note, HexDocs pointer.
2. Fit check and boundaries: good fits, HTML-to-PDF boundary, complex-script boundary, no PDF/A/PDF/UA claim, `pdfium-render` is not GUI-viewer proof.
3. One page per rendered gallery entry with image, job-to-be-done caption, module name, and manifest pointer.
4. Determinism proof: manifest, source PDF hashes, PNG hashes, renderer metadata, `manual.pdf` hash, and `mix rendro.launch_artifacts.check`.
5. Path/page-number proof: certificate frame or diagram plus footer pagination.

Manual visual tone:

- White document sheet, calm ink, sparse proof-oriented layout.
- Headings use ink-900; subheadings and rules may use blue-600 or teal-700.
- The manual may embed rendered PNG previews; this is acceptable because each source PDF is separately hash-checked. Do not imply PDF-in-PDF composition support.
- Do not expand the manual into a Prawn-style full reference catalog in this phase.

## Spacing Scale

Use the brand 4px base scale for Markdown/ExDoc styling:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Fine borders, tiny inline gaps |
| sm | 8px | Caption/hash adjacency, small image gaps |
| md | 16px | Paragraph and thumbnail group rhythm |
| lg | 24px | Section-local spacing |
| xl | 32px | Gallery entry separation |
| 2xl | 48px | Major README/guide section breaks |
| 3xl | 64px | Reserved for future landing pages, not README hero use |

Manual PDF point-scale equivalents:

| Token | Value | Usage |
|-------|-------|-------|
| manual-xs | 4pt | Path diagram/rule offsets |
| manual-sm | 8pt | Image/caption rhythm |
| manual-md | 12pt | Local proof grouping |
| manual-lg | 16pt | Cover/proof section breaks |
| manual-margin | 54pt | Page margin baseline |

Exceptions:

- README thumbnail width is fixed at `150`.
- Recipes guide image width is fixed at `320`.
- Existing preview raster dimensions remain manifest-driven: portrait entries `794x1123`, certificate `1123x794` at 96 DPI unless the manifest and docs contract are regenerated together.

## Typography

Markdown surfaces should respect GitHub/ExDoc host typography. Do not inject custom font loading just for this phase.

Brand typography for generated PDF/manual content:

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Manual body | 10.5-11pt | regular | renderer default or about 1.4 |
| Manual footer/caption | 9pt | regular/medium | renderer default or about 1.35 |
| Manual subheading | 13pt | medium/semibold | renderer default or about 1.25 |
| Manual heading | 20-26pt | semibold/bold | renderer default or about 1.2 |

Docs text hierarchy:

| Role | Contract |
|------|----------|
| README tagline | One short sentence immediately after badges |
| README feature bullets | 5-8 bullets, short noun-led phrases |
| Gallery captions | One sentence, concrete document job |
| Hash labels | `Source PDF SHA-256`, `PNG SHA-256`, `SHA-256` |
| Code/module names | Inline code; no decorative monospace outside technical identifiers |

Avoid:

- Huge SaaS hero typography.
- Condensed/decorative/retro fonts.
- Monospace for prose.
- Letter-spacing tricks that split or overlap rendered text.

## Color

Use the Rendro brand palette. The launch artifacts should read as ink, paper, layout grid, and reliable software.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | sheet-000 `#FFFFFF` / paper-100 `#F7F3EA` | Document previews, manual pages, warm table header bands |
| Secondary (25%) | ink-900 `#101827`, ink-700 `#1F2937`, line-300 `#D8D2C3` | Text, captions, borders, dividers, path/frame lines |
| Accent (10%) | blue-600 `#2C6BED` | Links, manual proof rule, active proof labels |
| Technical accent (5%) | teal-700 `#0E7C76`, plum-700 `#6E3CB8` | Sparse diagrams or Elixir-adjacent accents only |
| Warning | amber-600 `#C78600` | Boundary/warning labels with dark ink text, not small body text |
| Error | red-700 `#C24132` | Drift/error messages only |
| Success | green-700 `#147A4B` | Verified output labels only |

Accent reserved for:

- README/HexDocs links as provided by host styles.
- Manual proof rule and optional path diagram accents.
- Status/boundary labels when text also carries the meaning.

Launch fixture table style:

| Property | Value |
|----------|-------|
| `borders` | `[:outer, :rows]` |
| `header_fill` | `{247, 243, 234}` |
| `border_style.color` | `{216, 210, 195}` |
| `border_style.width` | about `0.6` |

Do not use `borders: :all` for long Statement or Receipt/Report gallery tables unless a small, isolated manual proof needs dense-grid evidence.

## Document Preview Treatment

Preview images must be presented as real document sheets.

Light mode:

- Image background: `#FFFFFF`.
- Border: `1px solid #D8D2C3`.
- Shadow: `0 8px 24px rgba(16, 24, 39, 0.08)`.
- Border radius: `0-4px`; avoid card-like rounded previews.

Dark/system mode:

- Keep the actual PNG/PDF page white. Do not invert, tint, dim, or generate duplicate dark assets.
- Border may change to `#1F2937`.
- Shadow may change to `0 8px 24px rgba(0, 0, 0, 0.28)`.
- Use `@media (prefers-color-scheme: dark)` only for the frame treatment around images.

Interaction:

- Images are plain anchors to full-size PNG files.
- `manual.pdf` is a plain text link.
- No modals, lightboxes, hover zoom, hover lift, animation, or JavaScript.
- Do not remove host focus outlines. Do not add custom focus styles unless they meet WCAG 2.2 AA contrast and are visible in light and dark modes.

## Copywriting Contract

Use proof-backed, maintainer-plain language. These phrases are locked unless implementation requires a tighter equivalent.

| Element | Copy |
|---------|------|
| README tagline | `Native PDF layout for Elixir.` |
| First mention | `Rendro is an open-source, Elixir-native PDF layout library for Phoenix teams that need reliable PDFs without Chrome.` |
| README gallery lead | `These previews are rendered by Rendro from curated deterministic recipe fixtures.` |
| Required proof line | `Source PDFs and the self-rendered manual are byte-checked by the required docs contract.` |
| Advisory proof line | `PNG rasters are regenerated and hash-checked in the pinned pdfium-render advisory lane.` |
| Evidence boundary | `pdfium-render rasters are render proof, not GUI-viewer proof.` |
| Fixture disclaimer | `Launch fixtures may use opt-in table polish; canonical recipe defaults remain unchanged.` |
| Manual line | `Self-rendered manual: [manual.pdf](assets/rendro/manual.pdf)` |
| Drift error pattern | `Launch artifacts drifted: {artifact}. Run mix rendro.launch_artifacts.gen in the pinned pdfium environment, then re-run the required docs contract and advisory raster check.` |

Guide captions must be concrete and job-shaped:

- Invoice: `Standard invoice from Elixir data through the canonical Invoice recipe.`
- Branded Invoice: `Branded invoice with registered font and logo assets.`
- Statement: `Multi-page statement with carried-forward balances and running page numbers.`
- Receipt / Report: `Receipt recipe scaled into a multi-page tabular report.`
- Certificate: `Landscape certificate with a Path-backed, geometry-derived border frame.`

Avoid these claims:

- `GUI-viewer proof`
- `PDF/A compliant`
- `PDF/UA compliant`
- `works in every viewer`
- `full HTML/CSS rendering`
- `browserless viewer`
- `prepress SDK`
- `pixel-perfect HTML-to-PDF`
- `launch gallery shows default recipe styling`

## Accessibility Contract

Alt text:

- Every preview image must have non-empty, descriptive alt text stored in the manifest and emitted into README and guides.
- Alt text must name the document type and the visible proof, not just `screenshot`, `preview`, or `image`.
- Alt text should not include hashes; hashes are adjacent selectable text.
- Current manifest-level alt text shape is acceptable and should be preserved or improved.

Links:

- Image links target the same full-size PNG as their `src`.
- `manual.pdf` link text must include the file name.
- Hashes must be exposed as visible text, not image-only content.
- Relative paths must work in GitHub, HexDocs, and the Hex package.

Contrast and dark mode:

- Normal text must meet WCAG 2.2 AA contrast.
- Meaningful non-text indicators must meet at least 3:1 against adjacent colors.
- `line-300` on `paper-100` is decorative only; do not rely on it as the sole proof marker.
- Dark mode must not hide the white sheet edge or image focus outline.

Keyboard/focus:

- Do not add hover-only disclosure.
- Do not suppress focus rings on image or manual links.
- Do not add custom tabindex values.
- Do not make hashes available only on hover.

## Anti-Patterns

Do not implement:

- Browser chrome, Chrome icons, HTML webpage screenshots, or print-dialog visuals.
- Printer, copy-machine, prepress, Acrobat-red ribbon, CHILI/spice, 3D cube, or commercial viewer-SDK visual language.
- README hero composite, custom marketing banner, or large landing-page layout inside the README.
- Dark-mode duplicate document previews.
- Inverted/tinted PNG previews in dark mode.
- Hover zoom/lift effects, modals, lightboxes, or animation.
- GUI-viewer, compliance, accessibility, or every-viewer overclaims.
- Any required CI path that downloads pdfium or regenerates PNG rasters.
- Global recipe default changes made only to improve screenshots.
- Dense full-grid table styling on long Statement or Receipt/Report previews.
