## Options For Applying Phase 84 Table Polish

### Option A: Launch-Only Table Decoration Transform

Add private helpers in `Rendro.LaunchArtifacts` that traverse generated launch documents or sections and update `%Rendro.Block{content: %Rendro.Table{}}` with the launch style.

Shape:

```elixir
@launch_table_style [
  borders: [:outer, :rows],
  border_style: %{color: {216, 210, 195}, width: 0.6},
  header_fill: {247, 243, 234}
]
```

Then either:

- call canonical `Recipe.document(data, opts)` and map `doc.sections[*].content[*]`;
- or assemble from `page_template/1` plus `sections/2`, transform sections before adding them to the document.

Pros:

- no public recipe API change;
- canonical defaults stay byte-identical;
- all screenshot-specific styling lives in launch-artifact code;
- good fit for "curated deterministic recipe fixtures" wording.

Cons:

- mutating structs directly bypasses `Rendro.table/2` normalization unless done carefully;
- BrandedInvoice needs its font/logo registration preserved;
- Statement/Receipt have generated multi-page table blocks, so the traversal must handle every section/block.

This is the recommended default. It best preserves D-01/D-02 from `86-CONTEXT.md`.

### Option B: Narrow Inert-Default Recipe Option

Add a recipe option such as `table_opts:` or `launch_table_style:` to Invoice, BrandedInvoice, Statement, and Receipt, defaulting to `[]`, and merge it into each `Rendro.table/2` call.

Pros:

- uses the public table builder normalization path;
- easy to test per recipe;
- Statement/Receipt measurement can receive the same `table_opts` used for rendering.

Cons:

- creates a public-looking recipe styling surface primarily for screenshots;
- more files and docs affected;
- easier for users to infer that polished launch gallery output is canonical default behavior.

Use this only if the document traversal becomes brittle or much more complex than expected.

### Option C: Change Canonical Recipe Defaults

Do not plan this. It conflicts with `86-CONTEXT.md` D-02 and Phase 84's byte-identity/default-inert requirement.

## Nyquist Validation Architecture

Use a "smallest meaningful drift fails one check" validation model:

| Drift/Event | Required Static Detector | Advisory Raster Detector |
|---|---|---|
| Remove a gallery recipe | exact ID test and manifest shape | check cannot find entry |
| Change recipe PDF output | regenerated source PDF hash drift | regenerated PNG hash likely drifts |
| Manually edit a PNG | committed PNG file hash versus manifest | regenerated PNG hash drift |
| Change PNG renderer/pin | manifest renderer/pin shape and public copy | renderer version drift |
| Change manual content | regenerated manual hash drift | not needed |
| Change manual hash in docs by hand | generated block equality/hash publication tests | not needed |
| Remove an image from README/guide | generated block equality and alt/path tests | not needed |
| Remove alt text | alt-text tests | not needed |
| Drop assets from package | Hex package inclusion test | not needed |
| Add GUI/PDF-A/PDF-UA overclaim | public copy guard | not needed |
| Put pdfium into required CI | negative CI/mix alias guard | not needed |

This is enough Nyquist coverage if each GAL requirement has at least one required detector and any raster-specific proof has an advisory detector.

Recommended additional tests before completing the phase:

- `static_contract_errors/0 does not require pdfium`: set an invalid pdfium env/finder and assert static errors do not include pdfium lookup failures.
- `mix ci alias remains pdfium-free`: inspect `Rendro.MixProject.project()[:aliases][:ci]`.
- `raster-advisory remains advisory`: assert no `needs:` and `continue-on-error: true` near the job block, or rely on existing raster claims test if already present.
- `package includes every launch asset`: manifest, manual, and all five `gallery/*.png`.
- `generated blocks are exact`: already covered by `static_contract_errors/0`; keep this as the source of truth.

## Concrete Implementation Risks

- Hash churn is expected. Avoid regenerating until fixture/copy/manual changes are stable.
- PNG hash determinism is only trustworthy in the pinned CI/pdfium environment. Do not bless or require PNG regeneration from developer laptops.
- The manual currently depends on existing PNG files. Generation order already writes PNGs before manual; preserve that order.
- Launch-only table polish should not change recipe defaults or existing recipe docs examples.
- Direct struct traversal for table polish is simple but must be careful to preserve all section/block fields.
- BrandedInvoice visual quality needs manual inspection after regeneration.
- `mix hex.build` in a test can leave `rendro-1.0.0.tar`; clean it up.
- Guardrail docs and CI can drift: update `priv/guardrails/required_status_checks.json` when changing advisory commands.
- README/guide generated blocks should not be hand-edited after generator changes.

## Planning Recommendation

Plan Phase 86 as five small waves:

1. Guard and preserve the existing scaffold.
2. Add launch-only fixture polish.
3. Align README/guide/manual copy with the brand book.
