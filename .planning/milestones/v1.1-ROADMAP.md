# Milestone v1.1: Layout Authoring Maturity

**Status:** ✅ SHIPPED 2026-04-30
**Phases:** 18-24
**Total Plans:** 17

## Overview

`v1.1` turned Rendro from a credible pure-Elixir PDF engine into a credible document authoring base. The milestone introduced explicit page-template and region authoring, deterministic wrapped-text measurement, authored keep/break pagination semantics, truthful fit validation, stronger multi-page table behavior, structured diagnostics/proof surfaces, and canonical recipe-level guidance for Phoenix adopters.

## Milestone Summary

**Key accomplishments:**

1. Established explicit page templates, named regions, sections, and truthful fit validation as the stable document-layout contract.
2. Added deterministic wrapped-text measurement plus authored keep/break semantics so flow pagination reflects author intent instead of implicit heuristics.
3. Strengthened table layout into a business-document surface with deterministic geometry, repeated headers, and runtime-consumed split policy.
4. Promoted diagnostics and pagination proof into public, test-backed behavior through `render_with_diagnostics/2`, inspector output, and repaired verification artifacts.
5. Lifted the new authoring surface into invoice recipes, the Accrue adapter, the Phoenix example, and README guidance.

**Key decisions:**

- Keep the core pure and deterministic while extending the existing `build -> compose -> measure -> paginate -> render -> validate` pipeline.
- Model flow layout with explicit `PageTemplate` / `Region` / `Section` structs instead of hidden defaults or Phoenix-specific helpers.
- Treat diagnostics, traceability, and verification artifacts as part of the product contract, not documentation afterthoughts.
- Close historically shipped-but-underverified requirements through explicit later repair phases instead of rewriting history.

**Issues resolved:**

- Replaced implicit default flow-page assumptions with explicit template-backed geometry.
- Closed the authored table split-policy runtime gap that had left `LAY-10` historically incomplete.
- Closed the diagnostics verification and traceability chain for `OBS-05` and `QUAL-06`.

**Issues deferred:**

- First public Hex release readiness was promoted into live backlog scope as `Phase 999.1` rather than being forced into `v1.1` close.
- Typography, assets, and broader Unicode support remain deferred to the next milestone boundary.

**Technical debt incurred:**

- Historical milestone phase directories remain in `.planning/phases/` for auditability and will need later cleanup/archive if you want a smaller live planning surface.
