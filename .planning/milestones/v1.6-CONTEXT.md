# Milestone v1.6: Async Delivery and Artifact Operations

## Project Description
This milestone focuses on operationalizing Rendro's pure-Elixir layout engine for high-volume SaaS environments. It introduces async generation capabilities, artifact metadata (manifests), and "Do Now" ecosystem integrations (threadline, mailglass, accrue) through a strict Behavior-Driven Adapter Pattern, ensuring the core remains pure and deterministic.

## Why This Milestone
Rendro's pure-Elixir layout and typography core has been hardened in previous milestones. The next major adoption hurdle for SaaS teams is reliable PDF generation at scale—specifically, enqueueing jobs, storing resulting artifacts, and handing them off to delivery pipelines (like email) while maintaining a strict audit trail.

## User-Visible Outcome
SaaS operators can reliably enqueue thousands of deterministic document generation jobs in the background, view durable audit logs for each render attempt, and automatically attach the resulting PDFs to transactional emails or store them in long-term object storage.

## Completion Class
Operational / Integration. This milestone requires live integration with standard Elixir async backends (Oban) and external ecosystem packages (`threadline`, `mailglass`, `accrue`), prioritizing robustness in production scenarios.

## Final Integrated Acceptance
- A user can enqueue a complex document render via Oban, have the worker process it, store the resulting artifact in a mock S3 bucket, log the success/failure to `threadline`, and hand off the artifact to `mailglass` for email delivery—all without blocking the web request cycle.

## Architectural Decisions

### Strict Behavior-Driven Adapter Pattern
**Rationale**: To prevent "core contamination" (adding `ecto`, `oban`, or `aws` as hard dependencies in `mix.exs`), all side-effects and async integrations must live in `lib/rendro/adapters/` (as optional dependencies) or as separate packages/recipes. The core engine will only emit pure data (`Rendro.Artifact`).
**Alternatives Considered**: Baking Oban and S3 uploads directly into the core library, which would violate the pure-core DNA and force database/AWS dependencies on all users.

## Error Handling Strategy
Layout errors must be captured, strictly serialized, and returned as structured diagnostics within the `Rendro.Artifact` manifest. Async workers (Oban) must explicitly catch these structured errors and propagate them to the `Rendro.Audit` behavior (e.g., `threadline`) to prevent opaque async failures. Retry policies will be handled by the configured Oban worker, but deterministic render failures should generally not be retried infinitely.

## Risks and Unknowns
- **Core Contamination**: Accidentally leaking side-effects into the `Rendro.Core` pipeline.
- **Leaking Artifact State**: Re-rendering the same document asynchronously might yield a different PDF hash due to non-deterministic timestamps, requiring strict seed/timestamp controls.
- **Opaque Failures**: Complex layout engine errors might get truncated in standard Oban logs, hindering operator debugging.

## Existing Codebase / Prior Art
- `Rendro.Core`: Existing pure layout and typography engine.
- `Rendro.Error`: Will be enhanced for structured serialization.
- `Rendro.Document`: Will be updated to expose deterministic timestamps/seeds for exact reproducibility.

## Relevant Requirements
- Define pure data structures for generation results (`Rendro.Artifact`).
- Establish standard pattern for async worker boundaries (`Rendro.Worker`).
- Pluggable storage integrations (`Rendro.Storage`).
- Ecosystem Integrations: `threadline` (audit), `mailglass` (email), `accrue` (billing).

## Scope
**In Scope**:
- Artifact manifests and metadata structures.
- Oban worker optional adapter and recipes.
- `Rendro.Storage` behavior definition.
- `threadline` and `mailglass` integration adapters.
- `accrue` billing document recipes.

**Out of Scope**:
- Built-in AWS/S3 uploads (users will implement the `Rendro.Storage` behavior or use community adapters).
- Webhooks for render completion (should be handled by host app's Oban worker).
- Integrations with `rulestead`, `sigra`, or `lockspire` (deferred to future milestones).

## Technical Constraints
- The `rendro` hex package must remain pure Elixir (no Ecto, Oban, or AWS dependencies in `mix.exs` default deps).
- Integrations must be optional extensions or documentation recipes.
- Strict determinism must be maintained for exact PDF bytes across async retries.

## Integration Points
- **Oban (~> 2.17)**: Async job queue execution.
- **Threadline (latest)**: Structured audit trails for template changes and render outcomes.
- **Mailglass (latest)**: Transactional email delivery with PDF attachments.
- **Accrue (latest)**: Deterministic financial/billing document generation.

## Testing Requirements
- **Unit**: 100% coverage on new pure data structures (`Rendro.Artifact`).
- **Integration**: Verifying the Oban worker adapter correctly queues, executes, and persists artifacts using mock storage and audit adapters.
- **E2E**: A full pipeline test enqueueing a render, passing the resulting artifact through a storage mock, and verifying the `threadline` audit log matches expectations.

## Acceptance Criteria
- `Rendro.Artifact` struct provides hash, byte size, and diagnostics.
- `Rendro.Worker` behavior defined and optional `rendro_oban` adapter published/documented.
- `Rendro.Storage` behavior defined.
- `Rendro.Audit` behavior defined with a working `threadline` adapter.
- Documentation includes copy-pasteable recipes for `mailglass` attachment bridge and `accrue` billing templates.

## Open Questions
- **Storage Interface**: How complex should the `Rendro.Storage` behavior be? Just `put(artifact)` and `get(hash)`, or does it need streaming support for massive PDFs? Currently leaning towards simple binary handoffs for v1.6.
