# Milestone v1.7: Authoring Ergonomics & Declarative DSL Roadmap

## Vision
Transform Rendro's document authoring experience from a verbose, manual AST-wiring process into a highly ergonomic, declarative Elixir DSL. The DSL will prioritize idiomatic Elixir patterns (e.g., drawing lessons from Phoenix/HEEx or Ash) to provide a principle-of-least-surprise experience. This allows developers to build complex, deterministic PDFs rapidly while keeping the underlying rendering engine pure and isolated.

## Success Criteria
- Developers can define full PDF layouts (documents, pages, sections) using a concise declarative DSL.
- Existing verbose recipes (like `BrandedInvoice`) can be rewritten with the new DSL, reducing structural boilerplate by over 50%.
- Invalid DSL structures are caught at compile-time or immediate runtime with errors that explicitly point to the offending DSL line, rather than deep pipeline internals.
- A standard library of reusable layout components (grids, split columns) is available and easily consumed within the DSL.
- Output from the DSL compiles down to `Rendro.Artifact` AST exactly, maintaining 100% determinism.

## Key Risks
- **DSL Lock-in**: Designing an overly rigid DSL that cannot support advanced or dynamic layout requirements.
- **Traceability Loss**: If macro expansion obscures original file locations, layout validation errors will be harder to debug.
- **Engine Coupling**: Bleeding macro logic or DX concerns into the pure internal layout/measurement core.

## Proof Strategy
We will implement the DSL iteratively from the inside out. First, we will establish the core DSL primitives (Blocks and Text), making an autonomous, deep architectural decision on Macro vs Functional approaches. Once proven, we'll scale up to Pages and Regions, ensuring structural validation logic is in place. Finally, we will build standard components and refactor the existing recipes to prove real-world usability and code reduction.

## Verification Classes
- **Structural Equivalence**: Unit tests proving the DSL produces the exact same `%Rendro.Document{}` AST structures as manual instantiation.
- **Error Traceability**: Tests ensuring invalid DSL configurations raise exceptions containing the correct file/line number of the user's code.
- **End-to-End Rendering**: Validating that DSL-generated documents pass through the standard `Build -> Compose -> Measure -> Paginate -> Render` pipeline successfully.

## Definition of Done
- All slices completed and verified.
- The `Rendro.Recipes.BrandedInvoice` and `Rendro.Recipes.Invoice` are successfully refactored using the DSL.
- Comprehensive documentation and guides for the DSL are added to HexDocs.
- Code conforms to existing `mix format` and `mix credo --strict` standards.

## Requirement Coverage
- "Simplifies document creation for Phoenix SaaS engineers" (from EPIC.md).
- "Reduces app-specific layout glue in downstream apps" (from EPIC.md).

## Boundary Map
- `DSL Layer` → constructs `Rendro.Document` / `Rendro.PageTemplate` / `Rendro.Block`
- `Rendro.Document` → consumed by `Rendro.Pipeline`
- `DSL Validation` → blocks invalid execution before hitting `Rendro.Pipeline`
