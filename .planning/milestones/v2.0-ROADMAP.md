# Milestone v2.0: Signature Fields & External Signing Preparation

**Status:** In Progress
**Started:** 2026-05-06
**Phases:** 55-57
**Total Plans:** 6

## Overview

`v2.0` adds a narrow signing-preparation story to Rendro by extending the existing authored forms surface with unsigned signature fields, keeping the core writer deterministic, and handing final artifacts into explicit external-signing workflows. The milestone is intentionally scoped to field authoring, signing preparation, and truthful support boundaries; cryptographic signing and compliance claims remain deferred.

## Phases

### Phase 55: Signature Field Authoring Contract

**Goal:** Extend Rendro's authored form surface with truthful unsigned signature-field semantics and validate-stage scope boundaries.
**Depends on:** Phase 54
**Plans:** 2 plans

Plans:

- [x] 55-01: Public authored signature-field API and validation taxonomy
- [x] 55-02: Unsigned signature-field fixtures, deterministic defaults, and regression coverage

**Requirements:** `SIGN-01`, `SIGN-02`
**Success criteria:**
1. Engineers can author an unsigned signature field through the supported public API.
2. Unsupported signature-field states fail with typed validate-stage errors before render.
3. The new field surface reuses the existing form model or proves a narrowly justified replacement without creating a second rendering path.

### Phase 56: Writer and External Signing Preparation Seam

**Goal:** Serialize deterministic signature-related PDF structures and add an artifact-first handoff seam for external signing workflows.
**Depends on:** Phase 55
**Plans:** 2 plans

Plans:

- [x] 56-01: Writer support for signature widget and related AcroForm structures
- [x] 56-02: Artifact-first signing-preparation API, final-byte handoff rules, and optional-adapter boundary

**Requirements:** `SIGN-03`, `PREP-01`, `PREP-02`, `PREP-03`
**Success criteria:**
1. Identical authored inputs produce deterministic unsigned signature-field structures.
2. A rendered `%Rendro.Artifact{}` can be prepared for external signing without changing `Rendro.render/2` semantics.
3. The signing-preparation seam consumes final artifact bytes and does not pull key custody or signer-specific trust material into core APIs.

### Phase 57: Support Contract and Proof Closure

**Goal:** Publish truthful signature-preparation boundaries and close the milestone with structural proof, docs-contract coverage, and evidence discipline.
**Depends on:** Phase 56
**Plans:** 2 plans

Plans:

- [ ] 57-01: Support-matrix, guide wording, and docs-contract updates for signature preparation
- [ ] 57-02: Structural fixtures, viewer-proof posture, and milestone verification notes

**Requirements:** `TRUST-01`, `TRUST-02`, `TRUST-03`
**Success criteria:**
1. Public support docs distinguish signature fields and signing preparation from unsupported digital-signature and compliance claims.
2. Signature-related viewer rows remain `unverified` unless a recorded checklist promotes them.
3. Verification artifacts prove structural correctness while keeping viewer and cryptographic validity claims separate.

## Phase Ordering Rationale

- Phase 55 constrains public semantics first so later writer work only handles supported field shapes.
- Phase 56 follows because signing-preparation workflows depend on final rendered bytes and deterministic object wiring.
- Phase 57 closes the milestone because this surface is trust-sensitive and easy to overclaim without dedicated proof and docs discipline.

## Deferred

- In-core cryptographic signing
- First-party certificate or key-management workflows
- PAdES/LTV/TSA/OCSP/CRL proof and compliance narratives
- Broad viewer support claims without recorded evidence
