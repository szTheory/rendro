# Milestone v1.9: Embedded Artifact Surfaces

**Status:** ✅ SHIPPED 2026-05-06
**Phases:** 48-50
**Total Plans:** 8

## Overview

`v1.9` extends Rendro's deterministic authored PDF surface with document-level embedded files and curated link annotations, while explicitly deferring native encryption, digital signatures, and broad annotation/review workflows. The shipped milestone keeps the public contract narrow and proof-backed: every claim in `priv/support_matrix.json` and `guides/api_stability.md` is now backed by structural proof through the existing Poppler lane and, where promoted, by recorded manual viewer evidence in Adobe Acrobat Reader and Apple Preview.

## Phases

### Phase 48: Embedded File Core Surface

**Goal**: Add a deterministic authored boundary for document-level embedded files and serialize it into the PDF catalog/object graph without widening into page-level review features.
**Depends on**: Phase 47
**Plans**: 2 plans

Plans:

- [x] 48-01: Document-owned embedded file registration + validate-stage rejection
- [x] 48-02: Deterministic embedded-file streams, file specs, and catalog `/Names`/`/AF` wiring

**Details:**
Phase 48 added `Rendro.EmbeddedFileRegistry` so documents own embedded file bytes and explicit metadata as pure data, exposed `Rendro.register_embedded_file/4` as the public builder surface, and routed metadata validation through `Rendro.Pipeline.Validate` with typed tuple errors instead of registration-time exceptions. The writer then gained deterministic embedded-file allocation, file-spec serialization, and catalog-level `/Names`, `/EmbeddedFiles`, and `/AF` wiring that appears only when embedded files exist and stays sorted by stable authored keys. Page-level file-attachment annotations were intentionally excluded.

### Phase 49: Curated Link Annotation Surface

**Goal**: Add the narrowest useful non-form annotation surface through deterministic external-URI and internal-destination links, reusing the existing `/Annots` writer seam.
**Depends on**: Phase 48
**Plans**: 3 plans

Plans:

- [x] 49-01: Explicit `Rendro.link/2` authored boundary + validate-stage curated link semantics
- [x] 49-02: Link measurement and pagination through the existing block pipeline
- [x] 49-03: Deterministic `/Link` annotation serialization from the writer

**Details:**
Phase 49 introduced `%Rendro.Link{content, target}` and `Rendro.link/2` accepting exactly one explicit target variant — `uri:` (`http`/`https` only) or `page:` (in-document) — and rejected `%Rendro.FormField{}`-wrapped links, malformed URIs, and out-of-range page targets in `Rendro.Pipeline.Validate`. Measurement reuses the existing `%Rendro.Block{}` recursion and preserves the link wrapper through fragmentation. The writer emits one annotation object per fragment in the page `/Annots` array, with external links serialized as narrow `/A << /S /URI /URI (...) >>` and internal links serialized as direct `/Dest [page_ref /Fit]` arrays. No named destinations, no `/GoToR`, no `/Launch`, no `/JavaScript`.

### Phase 50: Support-Boundary and Proof Closure

**Goal**: Close the milestone with truthful docs, support-matrix updates, and proof artifacts (structural + recorded manual viewer evidence) for the embedded-artifact surface.
**Depends on**: Phase 48, Phase 49
**Plans**: 3 plans

Plans:

- [x] 50-01: Family-first matrix + canonical guide wording + new docs-contract lane
- [x] 50-02: Representative embedded-artifact fixture + Poppler structural lane + validation document with explicit proof-lane split
- [x] 50-03: Manual viewer-evidence checkpoint + post-proof contract sync

**Details:**
Phase 50 extended `priv/support_matrix.json` with `embedded_files` and `links` as siblings of the existing `forms` family, added a new `Embedded artifact semantic-claims` lane to `mix docs.contract`, and published canonical wording in `guides/api_stability.md` that distinguishes PDF-internal embedded files from delivery-adapter attachments. Plan 02 committed `Rendro.Test.EmbeddedArtifactSupportFixture` — a deterministic two-page fixture exercising one embedded file plus external URI and internal page links in one document — and validated it through Poppler. Plan 03 recorded manual viewer evidence and promoted only the proof-backed pairs: Adobe Acrobat Reader is `supported` for both embedded files and links, Apple Preview is `supported` for links and `unverified` for embedded files (because the gap is on the viewer side, not on Rendro's authoring per D-09).

## Milestone Summary

**Decimal Phases:**

- None.

**Key Decisions:**

- Split embedded artifact surfaces from encryption and signatures into separate milestones instead of bundling them as one trust-surface release.
- Treated document-level embedded files as the primary next surface; not page-level file-attachment comments.
- Limited the `v1.9` annotation surface to curated links only — `http`/`https` URIs and in-document page targets — with no named destinations, no `/GoToR`, no generic action dictionaries.
- Extended the existing family-first nested support matrix with `embedded_files` and `links` as siblings of `forms` instead of introducing a generic `surfaces` wrapper or BCD-style per-leaf objects.
- Held viewer claims at `unverified` until manual evidence was recorded, then promoted only the proof-backed pairs.

**Issues Resolved:**

- Rendro can now author document-level embedded files with explicit, deterministic metadata.
- Rendro can now author deterministic external-URI and internal-page links and serialize them as `/Link` annotations through the existing page `/Annots` seam.
- The published support contract for the new families is one canonical truth source backed by structural proof and (where promoted) recorded manual viewer evidence.
- A representative fixture exercises the combined surface in a single document and is part of both the structural proof lane and the manual viewer proof lane.

**Issues Deferred:**

- Apple Preview × `embedded_files` remains `unverified` (Preview's UI does not surface document-level embedded files; Rendro's authoring is correct per the structural lane).
- Native PDF encryption and digital-signature claims remain future-milestone scope (v1.10 and v2.0 respectively).
- Generic annotations, comments, highlights, stamps, and review workflows remain explicitly out of scope.
- Page-level file-attachment annotations remain unsupported by deliberate boundary choice.

**Technical Debt Incurred:**

- `49-VERIFICATION.md` was never generated — the phase is functionally verified by `50-VERIFICATION.md`'s green `mix test` lane and the gsd-integration-checker E2E trace, but the per-phase verifier report is missing. Remediable retroactively.
- `49-VALIDATION.md` and `50-VALIDATION.md` frontmatter both still show `wave_0_complete: false` despite all targeted tests being green. Tracking-artifact only; no functional impact.
- SUMMARY frontmatter shape is inconsistent across phases (only 50-03 carries the explicit `requirements:` list; other plans either map requirements through PLAN.md or omit the field). Worth standardizing for the next milestone.

**Audit Status:** `tech_debt` (per `v1.9-MILESTONE-AUDIT.md` 2026-05-06) — all 7 requirements satisfied, all cross-phase wiring verified end-to-end, full test suite + docs gate green. The accumulated debt is purely documentation/tracking-artifact in nature and was accepted at milestone close.
