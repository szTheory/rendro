# Milestone v2.1: Cryptographic Signing & Signed-Artifact Proof

**Status:** ✅ SHIPPED 2026-05-07
**Started:** 2026-05-07
**Phases:** 60-63
**Total Plans:** 8

## Overview

`v2.1` adds one proof-backed cryptographic-signing path to Rendro by extending the shipped unsigned/preparation seam into actual signing, keeping the core writer pure, and proving the resulting signed-artifact posture through optional validation tooling. The milestone is intentionally scoped to cryptographic signing, signed-artifact proof, and truthful support boundaries; long-lived signatures, viewer promotion, and compliance claims remain deferred.

## Phases

### Phase 60: Public Cryptographic-Signing Contract

**Goal:** Lock the public `Rendro.Sign.sign/2` contract, redaction rules, and signed-artifact metadata posture without widening the shipped unsigned/preparation boundary.
**Depends on:** Phase 59
**Plans:** 2 plans

Plans:

- [x] 60-01: Signing API option contract, field validation, and typed error taxonomy
- [x] 60-02: Signed-artifact metadata, redaction coverage, and deterministic-vs-signed behavior proof

**Requirements:** `SIGN-04`, `SIGN-05`, `SIGN-06`
**Success criteria:**
1. Engineers can sign a rendered artifact through one explicit public API without changing `Rendro.render/2` semantics.
2. Invalid field selection and malformed adapter configuration fail with typed, redacted errors.
3. Signed artifacts clearly advertise non-deterministic signing state and do not leak secret material in shared metadata.

### Phase 61: First-Party Signing and Validation Adapters

**Depends on:** Phase 60
**Plans:** 2 plans

## Phase Ordering Rationale

- Phase 60 constrained public semantics first so adapter work could not smuggle wider trust claims into the API.
- Phase 61 followed because signing and validation adapters were the first concrete proof-backed implementation boundary for the milestone.
- Phase 62 closed the support contract only after the real toolchain path existed, keeping public claims downstream of evidence.
- Phase 63 closed the milestone with explicit verification and operational guidance so the signed-artifact story is auditable rather than implied.

## Deferred

- In-core key custody, certificate-store management, or HSM orchestration
- Timestamp, revocation, and long-lived-signature evidence
- PAdES/LTV/TSA/OCSP/CRL and broad compliance narratives
- Broad viewer support claims without recorded evidence

## Milestone Summary

**Key Decisions:**

- Kept cryptographic signing artifact-first over the shipped unsigned/preparation seam instead of reopening render-time semantics.
- Kept signing credentials, executable invocation, and tool-specific metadata adapter-local and optional at runtime.
- Treated signed output as intentionally non-deterministic while preserving deterministic claims for authored unsigned render output.
- Separated cryptographic signature integrity, certificate trust, viewer posture, and compliance narratives in every public surface.
- Required the `signing-live-proof` job to exist both as executable proof and as an enforced GitHub branch-protection gate.

**Issues Resolved:**

- Added the public `Rendro.Sign.sign/2` contract with typed, redacted sign-stage failures and explicit prepared-artifact rejection.
- Added first-party optional pyHanko and pdfsig adapters that keep runtime-executable and integrity-vs-trust boundaries explicit.
- Added a live proof lane with checked-in non-secret signing fixtures and a dedicated CI job.
- Aligned `priv/support_matrix.json`, `guides/api_stability.md`, `guides/integrations.md`, docs-contract tests, and verification artifacts around one exact supported signing path.
- Closed the milestone audit gap by adding Phase 60, 61, and 62 verification artifacts and syncing the full requirement traceability chain.

**Issues Deferred:**

- Long-lived signatures, timestamps, revocation evidence, and PAdES/LTV/TSA/OCSP/CRL remain deferred beyond `v2.1`.
- Signature-specific viewer promotion remains `unverified` until exact per-viewer evidence is recorded.
- Additional signing adapters beyond the pyHanko/pdfsig proof-backed path remain future work only if demand and evidence justify them.

**Technical Debt Incurred:**

- The validation adapter seam is still function-based (`function_exported?/3`) rather than backed by a dedicated behaviour contract.

**Audit Status:** `passed` — all 9 requirements satisfied, all 4 milestone phases closed, cross-phase signing/validation/docs flows verified, and the live proof gate is enforced. See `milestones/v2.1-MILESTONE-AUDIT.md`.
