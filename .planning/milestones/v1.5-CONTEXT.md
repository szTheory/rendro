# Milestone: Bulletproof Core & Feature Completeness (v1.5)

## Project Description
This milestone focuses on pushing the Rendro core rendering engine to full feature-completeness and "bulletproof" reliability. Per user guidance, we are explicitly avoiding external ecosystem integrations (e.g., Oban, Hex.pm adapters) at this stage. Instead, we are deepening the pure-Elixir layout capabilities—specifically targeting advanced table fragmentation, nested data layouts, edge-case pagination (widows/orphans), and structural validation of the generated PDF artifacts to guarantee operational trust.

## Why This Milestone
The user wants to ensure the PDF generation engine is "meaty, feature complete, and batteries included" before adopting it heavily or integrating it with other external libraries. By completing the advanced layout primitives and introducing structural validation, Rendro graduates from a "capable" PDF generator to a "bulletproof" enterprise-grade rendering engine that can be trusted with complex, dynamic SaaS reports.

## User-Visible Outcome
- Users can confidently render massive, multi-page data tables with nested elements and explicit cell fragmentation rules without layout distortions or silent truncations.
- Users receive structural validation reports alongside generated PDFs, proving that the output is not just visually correct but structurally sound.
- Users can define strict break semantics (widow/orphan controls) for critical text blocks and headers.

## Completion Class
Integration / Operational. This milestone touches the core rendering and layout pipeline and demands extensive deterministic and visual testing.

## Final Integrated Acceptance
- A comprehensive "Stress Test" report PDF (containing nested tables, long text blocks with widow constraints, and complex pagination) is generated deterministically.
- Structural validation runs against the generated stress-test PDF and reports zero structural errors.
- The pipeline gracefully handles and reports layout boundary violations instead of crashing or generating corrupt PDFs.

## Architectural Decisions
### Focus on Pure Core Features over Integration
*Rationale:* The user explicitly requested to hold off on integrating with other Hex.pm libraries. We will maintain the strict `rendro` pure-core boundary and focus our efforts exclusively on layout algorithms and internal diagnostics.

### Strict Table Fragmentation Semantics
*Rationale:* Tables will not rely on "best effort" auto-sizing for page breaks. We will enforce explicit fragmentation rules to maintain the deterministic DNA of the project.

## Error Handling Strategy
- **Layout Overflow:** If complex nested tables or constrained blocks cannot be paginated without violating rules, Rendro will emit a structured layout diagnostic rather than producing an overlapping or truncated PDF.
- **Validation Failures:** Structural PDF anomalies detected during the verification phase will be exposed as formal errors in the `Rendro.Error` pipeline.

## Risks and Unknowns
- **Nested Table Complexity:** Calculating deterministic heights for tables within tables across page breaks introduces significant algorithmic complexity and potential performance bottlenecks.
- **Validation Tooling:** Implementing structural PDF validation natively in Elixir without relying on external C-binaries (like poppler) may be limited in scope. We must define "validation" carefully.

## Existing Codebase / Prior Art
- `Rendro.Pipeline.Paginate` and `Rendro.Pipeline.Measure`: Existing measurement and pagination stages will require deep algorithmic enhancements for nested structures.
- `Rendro.Table`: Currently supports basic continuation but needs expansion for nested rows and advanced fragmentation.

## Relevant Requirements
- Epic v1.1 / v1.5 objectives: Robust Flow Enhancements and Validation/Trust Surfaces.
- Maintain deterministic layout and pagination guarantees.

## Scope
- **In Scope:** Nested table layout, advanced cell fragmentation, widow/orphan block controls, native structural validation checks (advisory lanes).
- **Out of Scope:** External ecosystem adapters (Hex.pm integrations), async worker integrations (Oban), full HTML/CSS auto-layout models.

## Technical Constraints
- Core rendering MUST remain pure Elixir.
- Output MUST remain 100% deterministic.
- Performance budgets must be maintained despite increased layout complexity.

## Integration Points
- Purely internal to the `rendro` core layout, measurement, and validation pipeline.

## Testing Requirements
- **Deterministic Testing:** Byte-for-byte or stable structural output assertions for all new layout features.
- **Stress Testing:** Introduction of a new "complex report" fixture to test the limits of the pagination engine.
- **Advisory Validation:** Tests to prove that the structural validation catches known bad PDF configurations.

## Acceptance Criteria
- [ ] Nested tables calculate height deterministically and fragment across pages accurately.
- [ ] Widow and orphan controls are respected during block pagination.
- [ ] A native or advisory structural validation pass can be executed on generated PDFs.
- [ ] All new features pass existing deterministic and visual contract tests.

## Open Questions
- How deep should the structural PDF validation go? Should we build a pure-Elixir cross-reference checker, or rely on visual snapshot proofs for "bulletproof" claims?
