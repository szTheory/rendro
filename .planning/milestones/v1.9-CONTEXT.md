# Milestone v1.9: Embedded Artifact Surfaces

## Project Description

This milestone extends Rendro's deterministic PDF generation model with narrowly-scoped embedded artifact surfaces inside the PDF itself. It focuses on document-level embedded files and curated link annotations, keeping the public API explicit, deterministic, and truthful about viewer behavior.

## Why This Milestone

Rendro already supports operational delivery attachments outside the PDF through adapters and deterministic interactive widgets inside the PDF through AcroForms. The next coherent step is to deepen the authored document surface without yet taking on the heavier trust-model changes of encryption or digital signing.

## User-Visible Outcome

Engineers can ship PDFs that contain embedded related artifacts and authored links while keeping support boundaries narrow and auditable.

## Completion Class

Product / Contract. The engine must emit deterministic embedded-artifact structures and publish a truthful support boundary for them.

## Final Integrated Acceptance

- A generated PDF can contain explicit document-level embedded files with deterministic metadata.
- A generated PDF can contain curated link annotations for external URIs and internal destinations.
- Public docs and support-matrix claims remain narrower than the actual PDF spec surface.
- Verification distinguishes structural proof from viewer discoverability or policy behavior.

## Architectural Decisions

### Embedded Files Before Broad Annotations

Rendro will model document-level embedded files first rather than generic annotations or review/comment workflows.

*Rationale:* Embedded files fit Rendro's document-authoring story and avoid opening the broader annotation/workflow surface too early.

### Curated Link Surface Only

If page-level annotations are included in this milestone, they will be limited to authored links.

*Rationale:* Links are the narrowest, least-surprising annotation type and reuse the existing `/Annots` seam already exercised by forms.

### Encryption and Signing Stay Deferred

Native PDF encryption and digital-signature claims remain out of scope for `v1.9`.

*Rationale:* Both surfaces change Rendro's trust model more drastically than embedded artifacts do and need their own milestone-level proof story.

## Error Handling Strategy

Invalid or ambiguous embedded-file and link configuration should fail before render with structured errors tied to the authored element or document-level option that caused the issue.

## Risks and Unknowns

- **Viewer discoverability:** Some viewers hide document-level embedded files behind an attachments pane.
- **Security policy variance:** Viewers may warn on or block certain embedded file types.
- **Terminology confusion:** "attachment" already means delivery attachment in some Rendro adapters, so naming must clearly distinguish embedded PDF artifacts from delivery concerns.

## Existing Codebase / Prior Art

- `Rendro.PDF.Writer` already emits `/Annots` for AcroForm widgets and provides the natural seam for curated non-form annotation support.
- `Rendro.Adapters.Mailglass` and `Rendro.Adapters.Phoenix` already cover delivery/download attachment flows outside the PDF binary.

## Relevant Requirements

- Core pure-Elixir rendering capability
- Deterministic output for authored inputs
- Truthful support boundaries and verification lanes

## Scope

**In Scope:**
- Document-level embedded files
- Explicit metadata for embedded files
- Curated link annotations (`uri` and internal destinations)
- Support-matrix and docs-contract updates for the new surface

**Out of Scope:**
- Generic annotations, comments, highlights, stamps, or review workflows
- File-attachment annotations on pages
- Native PDF encryption
- Digital signatures, PAdES, or compliance claims

## Technical Constraints

- Must remain a pure Elixir implementation in core.
- Must not introduce implicit timestamps or mutable metadata defaults.
- Must preserve deterministic output for identical authored inputs.

## Integration Points

- None required in core.
- Existing delivery adapters remain separate from embedded-artifact support.

## Testing Requirements

- Unit tests for embedded-file metadata validation and PDF object emission.
- Writer tests for names-tree/catalog wiring and curated annotation serialization.
- End-to-end proof that representative PDFs remain structurally valid.

## Acceptance Criteria

- Engine successfully writes PDFs with embedded files and curated links.
- Validation catches unsupported or ambiguous authored state before render.
- Public support claims exactly match proof artifacts and support-matrix entries.
