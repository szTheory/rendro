# Milestone v1.7: Authoring Ergonomics & Declarative DSL

## Project Description
This milestone focuses on dramatically improving the developer experience of creating PDF templates in Rendro. It introduces a cleaner, more declarative Elixir DSL and reusable layout templates (components) that allow developers to build complex documents without writing excessive boilerplate or dealing directly with raw AST nodes.

## Why This Milestone
After drifting toward external ecosystem integrations, we are course-correcting back to core PDF capabilities. Rendro's engine is pure and deterministic, but the current authoring surface requires too much manual structural wiring. Developers expect a library to be easily consumed with high-level primitives that handle common layout composition tasks effortlessly. This brings Rendro to production-readiness in terms of developer ergonomics.

## User-Visible Outcome
Developers can define entire PDF documents using a concise, macro-driven or clean function-based declarative DSL. They can reuse standard layout components (like standard headers, footers, and grid sections) out of the box, reducing the code required to define an invoice or statement by over 50%.

## Completion Class
Contract / Integration. The primary output is a stable, documented public API for authoring templates.

## Final Integrated Acceptance
- A developer can write a complete, multi-page invoice layout using only the new DSL and predefined components.
- The DSL correctly compiles down to the existing `Rendro.Artifact` AST without introducing any core determinism regressions.
- The output PDF visually matches the output generated by the old, verbose AST-wiring approach.

## Architectural Decisions
### DSL Layering
The new DSL will be built purely as a macro or functional wrapper layer on top of the existing `Rendro.Core` document AST. It will not alter the internal measurement, layout, or rendering pipelines. 
*Rationale:* Keeps the engine pure and deterministic while providing a better DX shell.

## Error Handling Strategy
The DSL must provide compile-time or immediate runtime validation of layout structure (e.g., catching unclosed tags or invalid nesting) before hitting the layout engine. Errors should explicitly point to the DSL line or block where the invalid structure was defined.

## Risks and Unknowns
- **DSL Lock-in:** Creating a DSL that is too rigid might prevent advanced use cases.
- **Traceability:** Errors in the core engine might be harder to map back to the high-level DSL constructs if we don't carry source location metadata through the AST.

## Existing Codebase / Prior Art
- `Rendro.Core` AST nodes (e.g., `Rendro.Document`, `Rendro.PageTemplate`, `Rendro.Block`).
- Existing `Rendro.Recipes` (like `BrandedInvoice`), which currently demonstrate the verbose wiring we want to replace.

## Relevant Requirements
- Simplifies document creation for Phoenix SaaS engineers.
- Reduces app-specific layout glue in downstream apps (from `EPIC.md`).

## Scope
**In Scope:**
- A new declarative DSL for creating documents, pages, and components.
- A library of canonical, reusable layout templates (grids, split columns).
- Updating existing recipes to use the new DSL.
- Documentation and guides for the new authoring experience.

**Out of Scope:**
- Changes to the underlying pagination, table splitting, or rendering engine.
- Adding HTML/CSS parsing.
- New ecosystem integrations.

## Technical Constraints
- Must remain pure Elixir.
- The DSL must produce 100% deterministic output identical to the manual AST construction.

## Integration Points
- None (purely an internal DX improvement layer).

## Testing Requirements
- Unit tests verifying the DSL translates exactly to expected AST structs.
- Integration tests ensuring the DSL-generated documents flow through the `Build -> Compose -> Measure -> Paginate -> Render` pipeline successfully.
- Docs-contract tests for all new DSL guides.

## Acceptance Criteria
- Developers can author documents using the new DSL.
- Existing verbose recipes are rewritten and simplified.
- Compilation/runtime layout errors point back to the DSL usage clearly.

## Open Questions
- Should the DSL be macro-based (e.g., like Phoenix HTML/HEEx or Ash resources) or purely function/piping-based?
