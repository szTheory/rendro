# Milestone v1.8: Interactive PDF Forms

**Status:** ✅ SHIPPED 2026-05-05
**Phases:** 45-47
**Total Plans:** 7

## Overview

`v1.8` expanded Rendro from static document generation into deterministic interactive PDF generation for the narrow AcroForm surface that the library can currently prove. The shipped milestone adds authored text fields, checkboxes, and radio groups to the existing core pipeline, keeps serialization deterministic through explicit appearance streams, and narrows public viewer claims to the support evidence actually recorded on disk.

## Phases

### Phase 45: AcroForm Text Field Foundation

**Goal**: Add the first interactive form-field surface to the core engine without creating a parallel rendering path.
**Depends on**: None
**Plans**: 2 plans

Plans:

- [x] 45-01: Form field domain, builder, validation, and measurement support
- [x] 45-02: AcroForm text field serialization

**Details:**
Phase 45 introduced `%Rendro.FormField{}` as the dedicated authored content node for interactive fields, wired `Rendro.form_field/3` into the public API, added authored-boundary validation for required names and supported values, and kept geometry on the existing measure/paginate seams. The writer then gained deterministic AcroForm catalog injection, widget annotations, and explicit normal appearance streams for text fields.

### Phase 46: Checkbox and Radio Button Widgets

**Goal**: Extend the same authored boundary to button widgets without widening the DSL or creating widget-specific layout machinery.
**Depends on**: Phase 45
**Plans**: 2 plans

Plans:

- [x] 46-01: Button widget contract and validation
- [x] 46-02: Button widget writer serialization

**Details:**
Phase 46 expanded `%Rendro.FormField{}` and `Rendro.form_field/3` to support `:checkbox` and `:radio` types, enforced explicit radio grouping and export values, and rejected contradictory default selections before render. The writer gained deterministic `/FT /Btn` serialization, checkbox on/off appearances, and grouped radio parent/child allocation with explicit exclusivity semantics.

### Phase 47: Form Validation and Viewer-Proof Closure

**Goal**: Close the milestone with truthful support boundaries, representative structural proof, and explicit viewer-status documentation.
**Depends on**: Phase 45, Phase 46
**Plans**: 3 plans

Plans:

- [x] 47-01: Supported widget and authored-state validation closure
- [x] 47-02: Docs and support-matrix contract closure
- [x] 47-03: Structural fixture and viewer-proof synchronization

**Details:**
Phase 47 hardened form validation against unsupported names, duplicate logical identity, invalid text metadata, duplicate radio export values, and contradictory defaults. It published the forms support contract in `priv/support_matrix.json`, locked public claims through docs-contract tests, and added a representative forms fixture to the Poppler lane. Apple Preview was recorded as supported for the named forms checklist; Adobe Acrobat Reader remains unverified and is not claimed as supported.

## Milestone Summary

**Decimal Phases:**

- None.

**Key Decisions:**

- Reused one `Rendro.form_field/3` / `%Rendro.FormField{}` authored boundary for all currently supported interactive widgets.
- Generated explicit appearance streams instead of relying on `NeedAppearances` so output remains deterministic.
- Treated viewer support as a proof-backed contract enforced by docs and `priv/support_matrix.json`.

**Issues Resolved:**

- Rendro can now author interactive text, checkbox, and radio widgets inside the normal deterministic pipeline.
- Invalid or contradictory form state fails before render instead of leaking into ambiguous PDF output.
- Public forms claims are now synchronized across docs, tests, and the machine-readable support matrix.

**Issues Deferred:**

- Adobe Acrobat Reader remains unverified for the forms viewer checklist and is intentionally excluded from supported claims.
- Signature fields, encryption, attachments, and annotations remain future milestone work.

**Technical Debt Incurred:**

- No core shortcuts were introduced, but the supported viewer surface remains intentionally narrow until more manual proof is recorded.
