# Milestone v1.8: Interactive PDF Forms

## Project Description
This milestone introduces core PDF engine support for rendering and filling interactive PDF forms. It enables Rendro to generate PDFs with embedded, fillable text fields, checkboxes, and radio buttons according to the PDF AcroForm specification, allowing developers to create dynamic documents directly from Elixir.

## Why This Milestone
Rendro is prioritizing deep, core PDF rendering capabilities over broad ecosystem integrations. Many enterprise use cases (e.g., contracts, applications, surveys) require generating documents that end-users can fill out directly. Adding form support solidifies Rendro's position as a "batteries-included" PDF platform without relying on external integrations.

## User-Visible Outcome
Developers can use Rendro primitives to define fillable form fields within their PDF templates. The resulting PDF files can be opened in supported viewers, and users can interactively fill in data.

## Completion Class
Operational / Contract. The engine must emit compliant PDF interactive form dictionaries.

## Final Integrated Acceptance
- A generated PDF containing form fields can be opened in Apple Preview, with other viewers only claimed when proof is recorded.
- The fields are correctly positioned and sized according to the template layout.
- Users can type text, toggle checkboxes, and select radio buttons within the generated PDF.
- Optional: Form fields can be pre-filled with data passed from Elixir during generation.

## Architectural Decisions
### AcroForm over XFA
We will target the standard AcroForm specification rather than the deprecated, XML-based XFA architecture.
*Rationale:* AcroForm is the universally supported standard for PDF forms and aligns cleanly with Rendro's deterministic binary generation.

### Field Definitions as Core Layout Nodes
Form fields will be introduced as first-class nodes in the `Rendro.Core` AST, seamlessly integrating into the existing measure, paginate, and render pipeline.
*Rationale:* Ensures form elements respect complex layout rules (e.g., flow, widows/orphans, nested tables).

## Error Handling Strategy
Invalid form field configurations (e.g., duplicate field names, invalid value types) should raise structured errors during the `validate` or `build` pipeline phases, pointing back to the specific element in the template.

## Risks and Unknowns
- **Viewer Compatibility:** Different PDF viewers handle form fields inconsistently (especially visual styling of the form field widget). We must test across a matrix of standard viewers.
- **Font Subsetting for Forms:** Interactive fields need access to fonts to render user input. We need to determine how to embed fonts cleanly for form fields without ballooning file size.

## Existing Codebase / Prior Art
- `Rendro.Core` AST nodes and pipeline.
- `Rendro.PDF.Writer` and `Rendro.PDF.Object` which will need to handle form dictionary output.

## Relevant Requirements
- Core pure-Elixir rendering capability.
- Support for complex business document workflows.

## Scope
**In Scope:**
- Text fields (single and multi-line).
- Checkboxes and radio buttons.
- Pre-filling forms with default values.
- Core AST additions for form elements.
- PDF Writer updates to emit the `/AcroForm` catalog and widget annotations.

**Out of Scope:**
- XFA (XML Forms Architecture) support.
- JavaScript inside the PDF for form validation or calculations.
- Digital signature fields (to be handled in a separate milestone).

## Technical Constraints
- Must remain a pure Elixir implementation.
- Form field annotations must be deterministically hashable (no random IDs or timestamps that change the bytes unless explicitly controlled).

## Integration Points
- None.

## Testing Requirements
- Unit tests verifying AcroForm catalog generation and widget annotation formatting.
- Integration tests ensuring form nodes flow through layout properly.
- End-to-end tests validating the generated PDF structure with `pdfinfo` or a custom parser.

## Acceptance Criteria
- Engine successfully writes a document with interactive form widgets.
- The fields respect the layout dimensions and positions specified in the template.
- Test suite covers edge cases like forms spanning pages or inside nested tables.

## Resolved Questions
- The default appearance (`/DA`) uses explicit Helvetica-compatible resources and deterministic authored appearance streams so viewers do not have to synthesize widget visuals.
