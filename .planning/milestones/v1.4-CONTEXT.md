# Milestone: Advanced Layout & Typography (v1.4)

## Project Description
This milestone focuses on advancing the core layout engine of Rendro to support complex business documents, specifically addressing advanced typography (i18n, font subsetting, shaping) and robust flow enhancements (complex tables, block-level break semantics like widow/orphan management) while strictly maintaining the deterministic pure-Elixir rendering constraint.

## Why This Milestone
To achieve the "batteries included" and "feature complete" vision for Elixir SaaS apps, Rendro must handle complex internationalized text and advanced, multi-page data reporting tables without layout errors. These capabilities allow Rendro to graduate from basic statements to supporting enterprise-grade, localized, and highly dynamic document structures.

## User-Visible Outcome
- Users can render documents containing diverse character sets and text directionality (i18n) without missing glyphs or improper shaping.
- Users can generate complex tables with nested data and explicit cell fragmentation across pages without silent truncation.
- Users can control text flow with advanced widow/orphan management to ensure professional document appearance.

## Completion Class
Integration / Operational. This milestone modifies the core deterministic rendering engine. Extensive edge-case testing against visual output and deterministic byte-output constraints is required.

## Final Integrated Acceptance
- A fully internationalized document (e.g., containing Arabic, CJK, and Latin characters) renders deterministically with correct shaping and font fallbacks.
- A multi-page table with complex column spans and nested data breaks correctly across pages, repeating headers as instructed.
- A document containing specific widow/orphan constraints honors those constraints during pagination without layout failure.

## Architectural Decisions
### Typography & i18n Strategy
The system must support advanced font subsetting and shaping entirely in Elixir. 
*Rationale:* To avoid relying on HarfBuzz or external C bindings, we will leverage or implement pure-Elixir font parsers and shapers that align with the deterministic requirement.

### Table Pagination Semantics
Tables must explicitly define cell fragmentation rules rather than auto-sizing to fit content.
*Rationale:* Rendro's core DNA requires explicit constraints. We will extend the `{:fixed, ...}` and `{:share, ...}` paradigms to handle multi-page fragmentation explicitly, rejecting ambiguous auto-sizing.

## Error Handling Strategy
- **Layout Failures:** If a table row or block cannot be fragmented and exceeds the region height, Rendro will emit a structured diagnostic layout error rather than silently truncating or overflowing.
- **Font Fallbacks:** If a required glyph is missing across all provided fonts, a structured warning is emitted and a deterministic placeholder (e.g., the `.notdef` glyph) is rendered.

## Risks and Unknowns
- **Pure-Elixir Text Shaping Complexity:** Implementing advanced text shaping (like OpenType layout features) in pure Elixir is highly complex and computationally expensive. The performance impact on the rendering pipeline must be strictly monitored.
- **Complex Table State:** Maintaining state across page breaks for tables with `rowspan` and `colspan` can introduce subtle non-deterministic bugs if not handled carefully.

## Existing Codebase / Prior Art
- `Rendro.Text` and `Rendro.Pipeline.MeasuredText`: Existing text measurement will need significant enhancement.
- `Rendro.Table`: Currently supports basic explicit columns and repeated headers. Needs expansion for complex fragmentation.
- `Rendro.Font` and `Rendro.PDF.FontParser`: Needs expansion for subsetting and advanced layout tables.

## Relevant Requirements
- Advance core engine to support global SaaS deployments (i18n).
- Advance reporting capabilities for complex financial/data tables.
- Maintain deterministic layout and pagination guarantees.

## Scope
- **In Scope:** Font subsetting, advanced text shaping (LTR/RTL), widow/orphan block controls, explicit complex table fragmentation.
- **Out of Scope:** Full HTML/CSS auto-sizing table models, bidirectional text editing (only rendering is in scope), external C-binding shapers like HarfBuzz.

## Technical Constraints
- Must remain pure Elixir (no NIFs or external binaries for core layout).
- Output must remain 100% deterministic given the same inputs.
- Policies (max_pages, timeout) must continue to bound execution time despite increased layout complexity.

## Integration Points
- This milestone is purely internal to the `rendro` core layout and measurement pipeline. No external system integrations (like Stripe or Mailglass) are included here.

## Testing Requirements
- **Deterministic Testing:** Existing and new test cases must assert exact byte-for-byte or stable structural output.
- **Visual / Flow Regression:** Edge-case documents must be added to the `docs.contract` or visual verification lanes to ensure correct shaping and table breaking.
- **Performance Budget:** Render times for complex tables and shaped text must be benchmarked against baseline to ensure acceptable throughput.

## Acceptance Criteria
- [ ] Font subsetting correctly embeds only used glyphs into the PDF.
- [ ] Complex shaping rules (e.g., ligatures, kerning) are applied correctly for supported scripts.
- [ ] Widow and orphan constraints are respected during `Rendro.Pipeline.Paginate`.
- [ ] Tables with cells spanning multiple rows/columns break predictably across pages.

## Open Questions
- What specific scripts and languages should be targeted for the initial i18n shaping release? (e.g., start with Latin extended + Arabic, or broader?)
- Can we leverage existing Elixir libraries for OpenType parsing, or must we build it entirely within Rendro?
